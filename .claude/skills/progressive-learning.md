# Progressive Learning Skill

## When to Use This Skill

Activate this skill when the user:

- Asks for explanation of programming concepts, algorithms, or code patterns
- Requests tutorials or learning materials for technical topics
- Needs step-by-step breakdowns of complex implementations
- Wants to understand how/why code works, not just what it does
- Requests learning paths or skill development guidance
- Asks to create educational content or documentation
- Needs progressive exercises that build on each other

## Triggers

**Educational Requests**:

- "Explain [concept]"
- "How does [algorithm/pattern] work?"
- "Teach me [technology]"
- "Create a tutorial for [topic]"
- "I don't understand [code/concept]"

**Learning Path Design**:

- "What should I learn to [goal]?"
- "How do I progress from [level] to [level]?"
- "What are the prerequisites for [skill]?"

**Exercise Creation**:

- "Create practice exercises for [topic]"
- "Give me problems to practice [skill]"
- "Build my understanding of [concept]"

## Quick Reference: 7-Phase Process

For rapid application, follow this condensed workflow:

### 1. Assess

Determine current knowledge level

- Ask about existing experience with related topics
- Identify knowledge gaps and misconceptions
- Establish baseline understanding

### 2. Decompose

Break complex topic into components

- Identify core concepts and supporting topics
- Separate essential from nice-to-know information
- Create a logical breakdown of the subject

### 3. Map Dependencies

Identify prerequisites and sequence

- Determine which concepts must come first
- Find prerequisite relationships between topics
- Document concept dependency chains

### 4. Design Progression

Create step-by-step path with difficulty scaling

- Order components from foundational to advanced
- Plan gradual increase in complexity
- Ensure each step builds on previous understanding

### 5. Provide Examples

Concrete examples at each level

- Use simple, relatable examples early
- Progress to realistic, domain-specific examples
- Include both positive and edge-case examples

### 6. Verify Comprehension

Checkpoints before advancing

- Create questions that test understanding
- Provide opportunities for hands-on practice
- Assess readiness before moving forward

### 7. Reinforce

Exercises that consolidate understanding

- Design practice problems that apply concepts
- Create review materials for key topics
- Build confidence through repetition and variation

**Quick Quality Checks**:

- Can learner complete each checkpoint?
- Is prerequisite knowledge identified correctly?
- Does difficulty scale appropriately?
- Are examples concrete and practical?
- Does each phase build logically on previous ones?
- Are learning outcomes clear at each step?

## How It Works

### 1. Knowledge Assessment Phase

**Before teaching, assess current understanding:**

- [ ] Identify learner's current knowledge level
- [ ] Determine existing related knowledge to build upon
- [ ] Identify gaps or misconceptions to address
- [ ] Adapt explanation depth appropriately

**Ask assessment questions:**
- "Have you worked with [prerequisite concept] before?"
- "What's your experience with [related technology]?"
- "What specifically is unclear about [topic]?"

### 2. Concept Breakdown Phase

**Decompose complex topics into digestible components:**

- [ ] Identify core concept (single most important idea)
- [ ] List prerequisites (what must be understood first)
- [ ] Break into 3-5 logical learning steps
- [ ] Sequence steps from simplest to most complex
- [ ] Map dependencies between concepts

**Structure breakdown as:**
```
Core Concept: [One sentence summary]

Prerequisites:
- [Prerequisite 1] - why it's needed
- [Prerequisite 2] - why it's needed

Learning Steps:
1. [Foundational concept] - master this first
2. [Building block 1] - builds on step 1
3. [Building block 2] - combines 1 + 2
4. [Advanced application] - integrates all concepts
5. [Edge cases/variations] - deepens understanding
```

### 3. Progressive Example Phase

**Provide working code with educational progression:**

**Example 1 - Minimal (Core Concept)**:
- Simplest possible working code
- Remove all non-essential complexity
- Focus on ONE concept only
- Heavily commented with "why" not "what"

**Example 2 - Basic (Add One Complexity)**:
- Build on Example 1
- Add ONE new concept/feature
- Explain what changed and why
- Show relationship to previous example

**Example 3 - Practical (Real-World)**:
- Realistic implementation
- Multiple concepts integrated
- Include error handling, edge cases
- Explain trade-offs and design choices

**Template for each example:**
```python
# Example [N]: [What this demonstrates]
# New concepts: [What's new compared to previous]
# Why this matters: [Real-world relevance]

[code with inline comments]

# Key takeaways:
# - [Learning point 1]
# - [Learning point 2]
```

### 4. Exercise Design Phase

**Create progressive exercises that build understanding:**

**Exercise 1 - Guided Practice**:
- Modify provided code in specific ways
- Success criteria: clear and testable
- Hints: point to relevant concepts from examples
- Expected difficulty: 5-10 minutes

**Exercise 2 - Structured Challenge**:
- Implement concept from scratch with requirements
- Success criteria: include edge cases
- Guidance: high-level steps, not solution
- Expected difficulty: 15-20 minutes

**Exercise 3 - Open-Ended Application**:
- Apply concepts to new problem domain
- Success criteria: functional requirements only
- Guidance: minimal (encourage exploration)
- Expected difficulty: 30-45 minutes

**Exercise template:**
```
Exercise [N]: [Title]
Difficulty: [Easy/Medium/Hard]
Time: [Estimated minutes]
Concepts practiced: [List from tutorial]

Task:
[Clear description of what to build/modify]

Requirements:
- [Specific requirement 1]
- [Specific requirement 2]

Success criteria:
- [How to know it's correct]

Hints:
- [Hint 1: if stuck on X, review concept Y]
- [Hint 2: common mistake to avoid]

Extension challenges (optional):
- [Harder variation 1]
- [Harder variation 2]
```

### 5. Understanding Verification Phase

**Ensure comprehension before moving forward:**

- [ ] Ask learner to explain concept in their own words
- [ ] Request code explanation (what does this line do and why?)
- [ ] Pose "what if" scenarios to test understanding
- [ ] Check for common misconceptions
- [ ] Verify ability to apply concept to new scenario

**Verification questions:**
- "Can you explain why we [design choice] instead of [alternative]?"
- "What would happen if we changed [X] to [Y]?"
- "How would you apply this to [different scenario]?"
- "What are the trade-offs between [approach A] and [approach B]?"

### 6. Learning Path Documentation Phase

**For multi-topic learning requests:**

- [ ] Create skill tree (prerequisites → advanced topics)
- [ ] Define milestones (checkpoints to verify progress)
- [ ] Estimate time per topic (realistic learning hours)
- [ ] Provide resources (docs, references, practice sites)
- [ ] Define mastery criteria (how to know you've learned it)

**Learning path template:**
```
Learning Path: [Goal]
Total estimated time: [Hours/weeks]

Phase 1: Foundations ([time estimate])
├─ Topic 1: [Name]
│  ├─ Prerequisites: [None or list]
│  ├─ Key concepts: [List]
│  ├─ Practice: [Exercises/projects]
│  └─ Mastery check: [How to verify]
└─ Topic 2: [Name]
   └─ ...

Phase 2: Building Blocks ([time estimate])
[Same structure]

Phase 3: Advanced Applications ([time estimate])
[Same structure]

Milestones:
- [ ] Milestone 1: [Description of capability achieved]
- [ ] Milestone 2: [Description]

Resources:
- [Official documentation links]
- [Practice platforms]
- [Community resources]
```

## Explanation Techniques

### Use Multiple Mental Models

Present concept through different lenses:

1. **Analogy**: "It's like [familiar concept]..."
2. **Visual**: "Imagine a [diagram/flow description]..."
3. **Concrete**: "For example, when you [specific scenario]..."
4. **Abstract**: "The general principle is [theoretical explanation]..."

### Progressive Disclosure

Reveal complexity gradually:

1. Start with simplest case (happy path, no edge cases)
2. Add one complexity at a time
3. Explain why each complexity matters
4. Show how professionals handle it

### Connect to Prior Knowledge

Bridge new concepts to existing understanding:

- "You already know [familiar concept]. This is similar, but [key difference]."
- "Remember how [previous topic] worked? This builds on that by [addition]."
- "This solves the limitation you saw in [earlier example] where [problem]."

### Avoid Common Pitfalls

**DON'T**:
- ❌ Dump full complexity upfront
- ❌ Explain "what" without "why"
- ❌ Skip prerequisites (assume too much knowledge)
- ❌ Use jargon without defining it
- ❌ Show only perfect code (hide learning process)
- ❌ Give solutions without ensuring understanding

**DO**:
- ✅ Build complexity incrementally
- ✅ Explain reasoning and trade-offs
- ✅ Verify prerequisites, teach if missing
- ✅ Define terms before using them
- ✅ Show evolution (wrong → better → best)
- ✅ Guide discovery through questions

## Educational Content Quality Standards

### Code Examples Must Have

- [ ] Correct syntax (actually runs)
- [ ] Inline comments explaining WHY (not what)
- [ ] Progressive complexity (simple → advanced)
- [ ] Error handling (show best practices)
- [ ] Realistic naming (not foo/bar unless teaching naming)

### Explanations Must Have

- [ ] Core concept in one sentence
- [ ] Why it matters (real-world motivation)
- [ ] How it works (mechanism explanation)
- [ ] When to use it (applicability context)
- [ ] Common mistakes to avoid
- [ ] Relationship to other concepts

### Exercises Must Have

- [ ] Clear learning objectives
- [ ] Specific success criteria
- [ ] Difficulty appropriate to concepts covered
- [ ] Hints for common sticking points
- [ ] Progressive challenge (easy → hard variations)

## Boundaries

### This Skill WILL

- Explain programming concepts with appropriate depth
- Create step-by-step tutorials with progressive examples
- Design exercises that build understanding systematically
- Map prerequisites and create learning paths
- Adapt explanations to learner's current level
- Verify understanding through questions and challenges
- Provide multiple explanation approaches (analogy, visual, concrete)
- Show code evolution (simple → complex, wrong → right)

### This Skill WILL NOT

- Complete homework or assignments without educational context
- Provide answers without explanation or learning opportunity
- Skip foundational concepts needed for true understanding
- Use advanced concepts before simpler ones are mastered
- Assume knowledge without verification
- Give "just make it work" solutions without teaching principles

### When NOT to Use

- User needs direct implementation help (not learning mode)
- Question is about debugging specific code (use systematic-debugging instead)
- User explicitly requests "just the answer, no explanation"
- Task is production code review (use requesting-code-review instead)

## Integration with Other Skills

- Use BEFORE implementation to teach concepts being applied
- Use AFTER code review to explain why changes are needed
- Use WITH brainstorming to teach design thinking process
- Invoke when user shows confusion or misconceptions about concepts

## Example Workflows

### Scenario 1: "Explain recursion"

1. **Assess**: "Have you worked with functions calling themselves before?"
2. **Break down**:
   - Core concept: Function calls itself with simpler input
   - Prerequisites: Understanding functions, base case concept
   - Steps: Base case → recursive case → call stack → practical use
3. **Progressive examples**:
   - Example 1: Countdown (simplest recursion)
   - Example 2: Factorial (mathematical recursion)
   - Example 3: Tree traversal (practical application)
4. **Exercises**:
   - Exercise 1: Modify countdown to count up
   - Exercise 2: Implement sum of list recursively
   - Exercise 3: Recursive file system traversal
5. **Verify**: "Explain why we need a base case. What happens without it?"

### Scenario 2: "Create learning path for React"

1. **Assess prerequisites**: "Do you know JavaScript? ES6 features?"
2. **Map dependencies**:
   - Phase 1: JS prerequisites (destructuring, arrow functions, promises)
   - Phase 2: React basics (components, props, state)
   - Phase 3: Hooks (useState, useEffect, custom hooks)
   - Phase 4: Advanced patterns (context, reducers, performance)
3. **Define milestones**:
   - Milestone 1: Build static component
   - Milestone 2: Build interactive form
   - Milestone 3: Build data-fetching app
   - Milestone 4: Build complex dashboard
4. **Provide resources**: Official docs, practice projects, community tutorials

### Scenario 3: "I don't understand async/await"

1. **Assess**: "Do you understand Promises? Synchronous vs asynchronous?"
2. **Identify misconception**: "What specifically is confusing?"
3. **Build from known**:
   - Review: Callbacks → Promises (what you already know)
   - New: async/await as Promise syntax sugar
   - Why: Readability (compare side-by-side)
4. **Show evolution**:
   - Example 1: Callback hell
   - Example 2: Same with Promises
   - Example 3: Same with async/await
   - Explain: They're equivalent, just different syntax
5. **Practice**: Convert Promise-based code to async/await
6. **Verify**: "When would you NOT use async/await?" (understanding trade-offs)

---

**Remember**: The goal is understanding, not memorization. Prioritize "why" over "what", and ensure learners can apply concepts to new scenarios, not just reproduce examples.
